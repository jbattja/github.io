Getting Started

In this section we will cover some of the basic information regarding our APIs.

1. REST​

We have organised our APIs around REST. We use resource-oriented URLs that accept form-encoded request bodies and return JSON-encoded responses. We use standard HTTP response codes, authentication, and verbs.

2. Authentication​

Our APIs use API Keys to authenticate your requests. The API Keys are essentially software-level credentials that allow a program to access your account without the need for providing your actual username and password to the software. It is all based off OAUTH 2.0 specification. Our API keys consist of Client_ID and Client_Secret.

KeyDescription
Client_IDThese are meant to identify your account with us uniquely. They can be safely used and published in client side applications like mobile apps, Web Pages, etc.
Client_SecretThese are strictly confidential and should only be stored in your own servers and be used for server to server communication. You should never share your Client_Secret to third-party.

Your API keys will be provided to you as part of your onboarding onto our platform.

3. Environments​

We provide 2 environments for you to integrate into.

  1. Production
  2. Sandbox

Our Sandbox is a Production-like environment with which you can test your integration before you go live. We recommend you do this. Any transaction that you perform against our sandbox will not affect your live data. We will not talk to financial network participants when you test against our sandbox. The base URIs to access these environments are also different.

EnvironmentBase URI
Productionhttps://api.openfabric.co
Sandboxhttps://sandbox.openfabric.co

4. Idempotency​

Idempotency ensures reliability and resilience against network failures that could lead to multiple retries of the same API call.

There are 2 strategies that we have deployed to ensure API idempotency.

For transactions already created in our system, there is a clearly defined transaction state machine. For example, we would not allow a transaction that has already been ‘approved’ by our customer to be re-approved again despite multiple retries. An ‘approved’ transaction however can be cancelled. This strategy ensures idempotency of an already created transaction.

For new transaction requests, we support the use of Idempotency-Key in the requests. For a request with a unique Idempotency-Key, we will consider the request is valid and will create a new transaction in our system. All Idempotency-Keys are active for 24-hrs (and considered non-expired).

All subsequent transaction requests with an Idempotency-Key that is the same as an existing, non-expired Idempotency-Key will be considered a duplicate request. We will simply replay the response of the original transaction request.

A transaction request with an ‘Idempotency-Key’ that is expired will be considered a valid request. This ‘Idempotency-Key’ will now be active for the next 24-hrs.

5. Call Backs and Retries​

OpenFabric sends push notifications as web-hooks to our customers. We expect our customers to acknowledge the receipt of these notifications with a 200 OK and use these acknowledgements as confirmation of delivery. We understand there may be temporary issues with your systems and you may not be able to acknowledge these notifications.

For notifications that are not acknowledged, we employ an exponential backoff mechanism to determine when we re-send these notifications. This means retries happen with an exponentially increasing wait time after every attempt, until we hit 20 retries.

We currently use an interval of 30 sec, exponentialRate of 1.6 and maxRetry of 20 in our exponential function. Note: We may update these values from time to time as needed.

This roughly works out to, 16 retries of a non-acknowledged notification in the first 24 hours, 2 retries in the next 24 hours, 1 retry in next 24 hours and 1 final retry in the next 24 hours.
